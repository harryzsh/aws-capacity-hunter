# 配额（抢 g6e / g7e 前必读）

抢 g6e.48xlarge / g7e.48xlarge 之前，**先把配额提够**，否则就算抢到 ODCR 也起不了实例。

## 提哪个配额

g6e、g7e 都属于 EC2 **G 系列**，**共用同一个**配额（不是各提一个）：

| 项 | 值 |
|----|----|
| 配额名 | **Running On-Demand G and VT instances** |
| Quota code | `L-DB2E81BA` |
| Service | `ec2`（Amazon EC2） |
| 单位 | **vCPU**（不是台数！） |
| 每台 g6e.48xlarge / g7e.48xlarge | **192 vCPU** |

**换算**：把所有 g6e + g7e 的台数**加在一起** × 192，就是要提到的 vCPU。

| g6e 台数 | g7e 台数 | 合计台数 | 至少需要 vCPU |
|----------|----------|----------|---------------|
| 0 | 4 | 4 | 768 |
| 10 | 20 | 30 | 5,760 |
| 12 | 8 | 20 | 3,840 |

> ⚠️ 配额是**分 Region** 的。你在哪个 region 抢，就在哪个 region 提。

## 方式一：Console 提（最简单）

1. 登录后打开 **Service Quotas** 控制台：`https://console.aws.amazon.com/servicequotas/`
2. 左侧 **AWS services** → 搜索 **EC2** → 点 **Amazon Elastic Compute Cloud (Amazon EC2)**
3. **右上角先把 Region 切到目标区**（如 `us-east-1`）——配额分区，切错区白提
4. 搜索框输入 **`Running On-Demand G and VT instances`** → 点进该配额
5. 右上角点 **Request increase at account level**（账户级申请）
6. 在 **Increase quota value** 填目标 **vCPU 数**（注意是 vCPU！= 192 × 所有 g6e+g7e 总台数）→ 点 **Request**
7. 跟踪审批：左侧 **Request history** 看状态。大额 GPU 申请常转人工，急的话去 **Support → Create case → Service limit increase** 补 case 说明用途催进度

## 方式二：CLI 提

```bash
# 1. 查当前 G/VT vCPU 配额
aws service-quotas get-service-quota \
  --service-code ec2 --quota-code L-DB2E81BA --region us-east-1

# 2. 申请提到目标 vCPU（示例：10 台 g6e + 20 台 g7e = 30 台 = 5760）
aws service-quotas request-service-quota-increase \
  --service-code ec2 --quota-code L-DB2E81BA \
  --desired-value 5760 --region us-east-1

# 3. 查看申请历史 / 状态
aws service-quotas list-requested-service-quota-change-history-by-quota \
  --service-code ec2 --quota-code L-DB2E81BA --region us-east-1
```

## 方式三：直接用脚本预检（推荐抢前跑一次）

脚本自带 `--check-quota`，读当前 G/VT 配额并按你的计划（所有 g6e+g7e 台数）判断够不够：

```bash
# 按总数计划检查
python3 grab_g7e_odcr.py --check-quota --counts g6e.48xlarge=10 g7e.48xlarge=20

# 按显式 per-(机型,AZ) 计划检查（自动汇总各格台数）
python3 grab_g7e_odcr.py --check-quota \
  --az-counts g7e.48xlarge@us-east-1b=5 g7e.48xlarge@us-east-1d=3 \
              g6e.48xlarge@us-east-1b=2 g6e.48xlarge@us-east-1d=10

# 输出示例：
#   G/VT On-Demand quota (Running On-Demand G and VT instances, L-DB2E81BA) in us-east-1: 768 vCPU
#     plan: 10 x g6e.48xlarge = 1920 vCPU
#     plan: 20 x g7e.48xlarge = 3840 vCPU
#     total needed 5760 vCPU vs current 768 -> INSUFFICIENT
```

`--check-quota` 只读、不抢、不计费。它额外需要 IAM 权限 `servicequotas:GetServiceQuota`（见 README 的 IAM 说明）。

## 两个一定要记住的事

1. **配额按 vCPU、按 G 系列汇总**——g6e 和 g7e 共用一个额度，填的是两者总台数 × 192，不是各填各的。
2. **配额批了 ≠ 抢得到容量**。配额只决定“允许你起多少”，真正能不能起还要看那个 AZ 有没有货——所以才要 `grab_g7e_odcr.py --watch` 死等 ODCR。
