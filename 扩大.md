# i4i 抢占（ODCR）前置配额扩大清单

> 场景：客户用「自管理 EKS nodegroup + 每 AZ 一个 ASG + open ODCR」的方式，预先抢占 i4i 容量（Prime Day 式 24/7 grab-as-available）。
> 本文列出**抢之前**就必须提交的配额扩大（Service Quotas / 工单），按 计算 / 存储 / 网络 拆分。

---

## 0. 最关键的一条（先看这里）

**i4i 没有独立配额。** i4i 属于 EC2 Standard 实例族（`A, C, D, H, I, M, R, T, Z` 里的 **I**），它占用的是这一个 vCPU 配额：

| 配额名 | 配额码 | 默认值 | 可调 |
|---|---|---|---|
| Running On-Demand Standard (A, C, D, H, I, M, R, T, Z) instances | `L-1216C47A` | **5 vCPU** | 是 |

AWS 官方明确写了创建 ODCR 的失败条件：
> *"The requested quantity exceeds your On-Demand Instance limit for the selected instance family. Increase your On-Demand Instance limit for the instance family and try again."*
> — [Create a Capacity Reservation](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/capacity-reservations-create.html)

**含义：ODCR 本身就占配额，不是等实例起来才占。** 只要你预留 N 颗 i4i，就需要 `Standard On-Demand vCPU 配额 ≥ N × 每颗 vCPU`，否则连 ODCR 都建不出来。所以这是**第一个、也是必须最先提**的扩大项。

参考：[EC2 instance type quotas](https://docs.aws.amazon.com/ec2/latest/instancetypes/ec2-instance-quotas.html) ·  [vCPU-based On-Demand limits](https://aws.amazon.com/blogs/compute/preview-vcpu-based-instance-limits/)

## 通用：怎么在 Console 提额（所有配额项都是这套流程）

> 下面每一节表格里都会给「点这里直接到该配额」的深链，点进去后从第 4 步开始即可。

1. 登录控制台，打开 **Service Quotas**：<https://console.aws.amazon.com/servicequotas/home>
2. 左侧导航选 **AWS services**（AWS 服务）。
3. 搜索框输入服务名（如 `EC2`、`VPC`、`EBS`），点进去。
4. 在配额列表里找到目标配额（可用配额码搜，如 `L-1216C47A`），点配额名进详情。
5. 点右上角 **Request increase at account-level**（账户级提升）。
6. 在 **Increase quota value** 填新值（必须大于当前值），点 **Request**。
7. 跟踪进度：左侧 **Dashboard** 或服务详情页的 **Request history**。小额通常秒批；**大额（几千 vCPU）会转 AWS Support 工单**，需要时间审核，急的话联系 TAM / 开 Support case 催。

> ⚠️ 注意：**确认右上角 Region 选对了**——配额是按 Region 的，提错区域等于白提。i4i 抢哪个区，就在哪个区提。
>
> 参考：[Requesting a quota increase](https://docs.aws.amazon.com/servicequotas/latest/userguide/request-quota-increase.html)

| 实例 | vCPU | 内存(GiB) | 本地 NVMe SSD | 网络带宽 | EBS 带宽 |
|---|---|---|---|---|---|
| i4i.large | 2 | 16 | 468 GB | 最高 10 Gbps | 最高 10 Gbps |
| i4i.xlarge | 4 | 32 | 937 GB | 最高 10 Gbps | 最高 10 Gbps |
| i4i.2xlarge | 8 | 64 | 1,875 GB | 最高 12 Gbps | 最高 10 Gbps |
| i4i.4xlarge | 16 | 128 | 3,750 GB | 最高 25 Gbps | 最高 10 Gbps |
| i4i.8xlarge | 32 | 256 | 7,500 GB (2×3,750) | 18.75 Gbps | 18.75 Gbps |
| i4i.12xlarge | 48 | 384 | 11,250 GB (3×3,750) | 28.12 Gbps | 28.12 Gbps |
| i4i.16xlarge | 64 | 512 | 15,000 GB (4×3,750) | 37.5 Gbps | 37.5 Gbps |
| i4i.24xlarge | 96 | 768 | 22,500 GB (6×3,750) | 56.25 Gbps | 56.25 Gbps |
| i4i.32xlarge | 128 | 1,024 | 30,000 GB (8×3,750) | 75 Gbps | 56.25 Gbps |
| i4i.metal | 128 | 1,024 | 30,000 GB | 75 Gbps | 56.25 Gbps |

> **注意**：i4i 的存储是**本地 NVMe（instance store）**，不是 EBS，本地盘不占任何存储配额、也不能单独扩。这正是 i4i 的卖点。网络/EBS 带宽是实例自带的硬上限，也不需要提配额。

**vCPU 估算例子**：要抢 20 颗 i4i.32xlarge = 20 × 128 = **2,560 vCPU**，那 `L-1216C47A` 至少要提到 2,560（建议再留 ~20% buffer 给系统/其他 Standard 实例 → 提到 3,000+）。

## 1. 计算（Compute）— 必提

| 项 | 配额名 / 码 | 默认 | 建议值 | 说明 |
|---|---|---|---|---|
| **On-Demand Standard vCPU** | [`L-1216C47A`](https://console.aws.amazon.com/servicequotas/home/services/ec2/quotas/L-1216C47A) | 5 | `目标实例数 × 每颗vCPU + buffer` | **最高优先级**。ODCR 和 On-Demand 实例都吃这个。✅官方核实可调 |
| Spot Standard vCPU（如做 Spot 兜底） | [`L-34B43A08`](https://console.aws.amazon.com/servicequotas/home/services/ec2/quotas/L-34B43A08) | 5 | 同上量级 | 只有当你 ASG 里挂了 Spot 兜底才需要。✅官方核实可调 |
| EKS 集群数 | [`L-1194D53C`](https://console.aws.amazon.com/servicequotas/home/services/eks/quotas/L-1194D53C) | 100 | 一般够 | 多集群架构才需关注。✅可调 |
| Auto Scaling Group 数 / region | EC2 Auto Scaling `Auto Scaling groups per region` | **500** | 每 AZ×每实例族 一个 ASG，数量多时确认 | ✅官方核实默认 500、可调（Service Quotas 或 Auto Scaling Limits 工单）。一般够。 |
| 启动模板 / 启动配置数 | EC2 `Launch templates` / ASG `Launch configurations per region`(200) | 启动配置默认 200 | 自管理 nodegroup 用启动模板(LT)，一般够 | 启动配置（旧）默认 200；启动模板数请控制台查实际值。一般不会是瓶颈。 |

**要点**：
- 配额是**按 Region**算的（ENI 例外，见网络节）。你把 ODCR 分散到 3 个 AZ，区域总 vCPU 需求不变 → `L-1216C47A` 要覆盖**全部 AZ 的总和**。
- 大额 vCPU 提升（几千核）AWS 可能不会秒批，需要走工单 / 找 TAM，**务必提前几天到一两周申请**，别等抢的当天。
- **⚠️ 自管理 nodegroup 不吃 EKS 节点配额**：文档场景是「自管理 EKS nodegroup（裸 ASG + EC2）」，所以 EKS 的 `Managed node groups per cluster`（默认 30，[`L-6D54EA21`](https://console.aws.amazon.com/servicequotas/home/services/eks/quotas/L-6D54EA21)）和 `Nodes per managed node group`（默认 450，[`L-BD136A63`](https://console.aws.amazon.com/servicequotas/home/services/eks/quotas/L-BD136A63)）**这两个限制对你不生效**——它们只管 EKS *托管* nodegroup。**别白提这俩。** 反过来如果将来改用托管 nodegroup，单组超 450 节点就要提 `L-BD136A63`。

## 2. 存储（Storage）

i4i 本地盘不占配额，但 EKS 节点仍需 **EBS 根卷**（boot volume），节点规模大时累计可观。

| 项 | 配额名 | 默认 | 建议 |
|---|---|---|---|
| gp3 总容量 (TiB/region) | [`L-7A658B76`](https://console.aws.amazon.com/servicequotas/home/services/ebs/quotas/L-7A658B76)（EBS） | **多数区 50，少数区 300** | 节点数 × 根卷大小，留余量 |
| io2 总容量 (TiB/region) | [`L-09BD8365`](https://console.aws.amazon.com/servicequotas/home/services/ebs/quotas/L-09BD8365) | **20**（默认偏低） | 仅用 io2 高性能 EBS 时，注意默认才 20 TiB |
| io2 IOPS 聚合 | [`L-8D977E7E`](https://console.aws.amazon.com/servicequotas/home/services/ebs/quotas/L-8D977E7E) | 100,000 | 仅用 io2 时 |
| EBS 快照数 / region | [`L-309BACF6`](https://console.aws.amazon.com/servicequotas/home/services/ebs/quotas/L-309BACF6) | 100,000 | 一般够 |

> ✅ 以上 EBS 配额码均已对照官方 [Quotas for Amazon EBS](https://docs.aws.amazon.com/ebs/latest/userguide/ebs-resource-quotas.html) 核实，全部 Adjustable=Yes。**注意 gp3 默认在多数区域只有 50 TiB**（少数新区 300 TiB），例如 200 个节点 × 100 GiB 根卷 = 20 TiB 还在 50 TiB 内，但若根卷大或节点多就会撞顶——**先在 Service Quotas 查当前实际值**（默认值≠已批值）。
>
> 参考：在 Service Quotas 控制台搜 **Amazon Elastic Block Store (Amazon EBS)**。

## 3. 网络（Network）— EKS 大规模最容易被卡

EKS 用 VPC CNI，**每个 Pod 占一个 VPC 子网 IP + ENI 上的辅助 IP**，大规模抢占时网络配额和子网 IP 是隐形瓶颈。

| 项 | 配额名 / 码 | 默认 | 建议 / 说明 |
|---|---|---|---|
| **VPC 数 / region** | [`L-F678F1CE`](https://console.aws.amazon.com/servicequotas/home/services/vpc/quotas/L-F678F1CE) | **5** | 经典拦路虎。到上限就建不了新 VPC → 复用现有 VPC，或提配额。 |
| **网络接口 ENI / region** | [`L-DF5E4CA3`](https://console.aws.amazon.com/servicequotas/home/services/vpc/quotas/L-DF5E4CA3) | 5,000 | ⚠️**按 AZ 算，不是按区**：每 AZ 5,000，3 个 AZ 实际可用 15,000。EKS + VPC CNI 大量消耗 ENI，节点多时算清楚。✅可调 |
| **子网 IP 地址空间** | （非配额，靠 CIDR 规划） | — | **最关键**。Pod 从子网取 IP，i4i.32xlarge 能跑很多 Pod。子网 CIDR 要够大（如 /18、/19），不够要加 secondary CIDR。 |
| Elastic IP / region | [`L-0263D0A3`](https://console.aws.amazon.com/servicequotas/home/services/ec2/quotas/L-0263D0A3) | **5** | 节点要公网 IP / NAT 出网时提。✅可调 |
| 子网数 / VPC | [`L-407747CB`](https://console.aws.amazon.com/servicequotas/home/services/vpc/quotas/L-407747CB) | 200 | 一般够。✅可调 |
| 安全组规则数 / SG | [`L-0EA8095F`](https://console.aws.amazon.com/servicequotas/home/services/vpc/quotas/L-0EA8095F) | 60 入+60 出 | 规则多时确认。⚠️**有隐藏乘积上限**：`每SG规则数 × 每ENI的SG数 ≤ 1000`，不能两个都顶满。✅可调（在乘积约束内） |
| NAT 网关 | （带宽/连接数自动扩） | — | 大量出网时关注连接数 & 跨 AZ 流量成本。 |

**要点**：
- **子网 IP 耗尽**是 EKS 大规模最常见的硬故障：节点起来了、Pod 调度不上去，因为子网没 IP 了。抢占前务必按「峰值 Pod 数」反推子网 CIDR。
- VPC 默认 5 个的限制：如果客户账号已有 5 个 VPC，**eksctl 建新 VPC 会直接失败**，只能复用现有非生产 VPC（参考内部 smoke-test 经验）。

参考：[Amazon VPC quotas](https://docs.aws.amazon.com/vpc/latest/userguide/amazon-vpc-limits.html) ·  [EKS endpoints and quotas](https://docs.aws.amazon.com/general/latest/gr/eks.html)

## ➕ 还有什么也要扩 / 确认（容易漏，但对 i4i 大机器很关键）

i4i 单台很大（32xlarge = 128 vCPU），光把 vCPU 配额提够、节点起来了还不行——下面几个**不是 Service Quotas 里那种「填个数字」的配额**，但同样会卡死你，必须一并处理：

| 项 | 类型 | 为什么 i4i 场景要管 | 怎么处理 |
|---|---|---|---|
| **每节点最大 Pod 数** | VPC CNI 配置（非配额） | 默认 max-pods 由 `ENI 数 × (每ENI IP数-1)+2` 决定，128 vCPU 的大机器默认可能只能跑 ~100-250 个 Pod，**机器买了跑不满**。 | 开 **Prefix Delegation**（`ENABLE_PREFIX_DELEGATION=true`），每 ENI 分 /28 前缀，Pod 密度翻数倍。大机器必开。 |
| **子网 IP（再强调）** | CIDR 规划（非配额） | 开了 prefix delegation 后每节点要的 IP 更多，子网更容易耗尽。 | 子网用 /18~/19，或加 secondary CIDR（100.64/10）。 |
| **ELB / NLB + 目标组** | Service Quotas（ELB） | 若服务对外，节点/Pod 多 → 目标数暴涨。ALB 每目标组默认 1000 目标、每 region ALB 数有限。 | 在 Service Quotas 搜 **Elastic Load Balancing** 确认 `Targets per Target Group`、`Application/Network LBs per Region`。 |
| **ECR 镜像拉取限流** | API 限流（非配额） | 几十上百个节点**同时**拉同一镜像，会触发 ECR throttling，节点卡在 ContainerCreating。 | 用 **ECR pull-through cache** / 镜像预热 / 大镜像分层；必要时找 Support 提拉取速率。 |
| **EC2 / ODCR API 限流** | API 限流（非配额） | 24/7 grab-as-available 循环反复调 `CreateCapacityReservation` / `RunInstances`，太密会 `RequestLimitExceeded`。 | 抢占脚本加**指数退避**重试，别死循环猛打 API。 |
| **CloudWatch Logs 摄入** | Service Quotas（按需） | 大量节点日志集中上报可能撞 `PutLogEvents` 限流。 | 仅当日志量大时关注，搜 **CloudWatch Logs** 配额。 |

> **一句话**：vCPU + 网络 + EBS 是「填数字」的配额；**max-pods（prefix delegation）、ECR 拉取、API 限流**是「改配置 / 加退避」的隐形天花板。大机器场景这三个必看，否则会出现「实例 Running 了但 Pod 上不来」。

## 4. 推荐执行顺序（抢之前）

1. **先提 `L-1216C47A`（On-Demand Standard vCPU）** —— 数值 = 目标 i4i 总 vCPU + 20% buffer。这是大额项，最先提、最可能要工单/TAM。
2. 同步检查 **VPC 数**（≥1 可用）、**ENI**、**EIP**、**子网 CIDR 容量**，不足就一起提 / 重新规划网段。
3. 检查 **EBS gp3 容量配额**够不够节点根卷总量。
4. 配额全部 `Approved` 后，再 **dry-run 建 open ODCR**（按 AZ 拆），确认能建出来不报「exceeds limit」。
5. 最后才是 scale ASG → 节点自动落入同 AZ 的 open ODCR。

> **核心提醒**：open ODCR 一旦创建就开始计费（不管实例起没起）。提配额是免费的、可提前做；建 ODCR 才花钱，留到临门一脚。

---

## 5. 总结一句话

> 抢 i4i 之前，**最该提的是 EC2「On-Demand Standard vCPU」配额（L-1216C47A，默认才 5 核）**，因为 i4i 走的是 Standard 共享配额，而且 ODCR 预留就要占配额、不提就建不出来。其次按 EKS 规模检查 **VPC 数、ENI、子网 IP 空间** 三个网络项，以及节点根卷的 **EBS 容量**。本地 NVMe、实例网络/EBS 带宽是 i4i 自带的，不用提。
